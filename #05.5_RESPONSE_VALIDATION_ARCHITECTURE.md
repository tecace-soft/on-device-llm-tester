## 2. Why This Architecture

### Phase 4a/4b 분리 — 왜 LLM-as-a-Judge를 뒤로 미루는가?

- **모델 정확도는 디바이스에 의존하지 않는다**: 응답 품질은 모델 + 양자화 포맷에 의해 결정됨. 동일 모델·동일 양자화면 S25든 S26이든 같은 답을 냄. 디바이스별로 정확도를 반복 측정할 실익이 없음
- **대체재가 이미 충분하다**: lm-evaluation-harness, HELM, OpenCompass 등 LLM 품질 평가 도구는 성숙함. 이 영역에서 경쟁하는 건 차별성이 낮음
- **우리의 차별점은 하드웨어다**: "실 모바일 기기에서 LLM 추론 성능을 자동 측정하고 디바이스 간 비교하는 E2E 파이프라인"은 시장에 부족함. 이 방향으로 전문성을 깊게 파는 게 프로젝트 가치를 극대화

### Deterministic 전용 — 왜 외부 API를 안 쓰는가?

- **비용 zero, 의존성 zero**: CI 환경에서 API 키 관리, rate limit, 장애 대응 불필요
- **재현성 100%**: 같은 입력이면 항상 같은 결과. LLM judge는 확률적
- **하드웨어 벤치마크의 연장선**: "이 디바이스에서 이 모델이 정상 동작하는가"를 확인하는 것. 모델 지능 측정이 아님
- **4b 확장 경로 확보**: `QUALITY_EVAL_ARCHITECTURE.md`가 이미 있으므로 LLM judge가 필요해지면 바로 구현 가능

### validation_status vs quality_score — 왜 스키마가 다른가?

- 기존 `QUALITY_EVAL_ARCHITECTURE.md`는 `quality_score REAL (0.0~1.0)` + `quality_verdict (pass/fail/partial)` 설계
- Phase 4a는 점수가 아니라 **통과/실패/경고의 검증 결과**. 0.7점 같은 연속값이 아님
- 4b 구현 시 `quality_*` 컬럼을 추가하면 됨. 4a의 `validation_*`과 공존 가능

## 3. Validation Checks

### 3.1 Sanity Checks (모든 응답에 적용)

| 체크 | 로직 | 결과 |
|------|------|------|
| **empty_response** | `len(response.strip()) == 0` | `fail` |
| **truncated** | `output_token_count >= max_tokens` (test_config에서 조회) | `warn` |
| **error_status** | `status == 'error'` | `skip` (평가 대상 아님) |
| **gibberish** | 응답의 80%+ 가 non-ASCII 또는 반복 토큰 | `warn` |

**truncation 감지 상세**: max_tokens는 test_config.json의 모델별 설정에서 가져옴. output_token_count가 max_tokens의 95% 이상이면 truncated로 판정. 이는 "모델이 답을 다 못 했다"는 신호이므로 성능 메트릭 해석 시 주의 필요.

**gibberish 감지**: SLM이 붕괴(degenerate) 시 무의미한 토큰을 반복 생성하는 경우가 있음. 예: "the the the the..." 또는 깨진 UTF-8. 이를 감지하여 `warn` 처리.

```python
def check_gibberish(response: str) -> bool:
    """응답이 의미 없는 반복 토큰인지 감지."""
    if not response.strip():
        return False  # empty는 별도 체크

    words = response.split()
    if len(words) < 3:
        return False

    # 반복 토큰 비율 체크: 전체 단어 중 가장 빈번한 단어가 50%+ 차지
    from collections import Counter
    counts = Counter(words)
    most_common_count = counts.most_common(1)[0][1]
    if most_common_count / len(words) > 0.5:
        return True

    return False
```

### 3.2 Deterministic Eval (카테고리별)

| category | eval_strategy | 로직 | ground_truth 형태 |
|----------|--------------|------|-------------------|
| `math` | `deterministic` | 숫자 추출 + EM (tolerance ±0.1%) | 정확한 숫자 (`"722"`) |
| `minimal_output` | `deterministic` | 숫자/yes-no 매칭 | 단답 (`"4"`, `"yes"`) |
| `knowledge` | `deterministic_with_fallback` | 정답 키워드 containment | 핵심 단어 (`"Paris"`) |
| `reasoning` | `deterministic_with_fallback` | 정답 키워드 containment | 핵심 답 (`"A"`) |
| `structured_output` | `structural` | valid JSON/markdown 파싱 | 예상 구조 |
| `code` | `structural` | Python 구문 유효성 (`ast.parse`) | 없음 |
| `summarization` | `none` | sanity check만 | 없음 |
| `long_generation` | `none` | sanity check만 | 없음 |
| `creative` | `none` | sanity check만 | 없음 |

**eval_strategy 설명**:
- `deterministic`: ground_truth와 정확히 비교. 정답이 명확한 카테고리
- `deterministic_with_fallback`: containment 매칭. 정답이 응답에 포함되면 pass. 불포함이면 `uncertain` (LLM judge 없이는 판정 불가)
- `structural`: 응답의 구조적 유효성만 체크 (JSON valid? Python parseable?)
- `none`: sanity check만 적용. deterministic 판정 불가능한 카테고리

### 3.3 Math Eval

```python
import math
import re


def eval_math(response: str, ground_truth: str) -> tuple[str, str]:
    """Returns (status, detail). status: 'pass' | 'fail'."""
    try:
        expected = float(ground_truth)
    except ValueError:
        return "fail", f"Invalid ground_truth: {ground_truth}"

    numbers = re.findall(r'-?\d+\.?\d*', response)
    if not numbers:
        return "fail", "No numeric value found in response"

    for num_str in reversed(numbers):
        num = float(num_str)
        if math.isclose(num, expected, rel_tol=1e-3, abs_tol=1e-9):
            return "pass", f"Correct: {num_str} matches expected {ground_truth}"

    closest = min(numbers, key=lambda n: abs(float(n) - expected))
    return "fail", f"Incorrect: closest value {closest}, expected {ground_truth}"
```

### 3.4 Factoid/Knowledge Eval

```python
import re


def eval_containment(response: str, ground_truth: str) -> tuple[str, str]:
    """Returns (status, detail). status: 'pass' | 'uncertain'."""
    def normalize(text: str) -> str:
        text = text.lower().strip()
        text = re.sub(r'\b(the|a|an)\b', '', text)
        text = re.sub(r'[^\w\s]', '', text)
        text = re.sub(r'\s+', ' ', text).strip()
        return text

    norm_truth = normalize(ground_truth)
    norm_response = normalize(response)

    if norm_truth in norm_response:
        return "pass", f"Ground truth '{ground_truth}' found in response"

    if norm_response in norm_truth and len(norm_response) > 2:
        return "pass", f"Response is subset of ground truth"

    return "uncertain", f"Ground truth '{ground_truth}' not found — needs LLM judge (Phase 4b)"
```

### 3.5 Structural Eval

```python
import ast
import json


def eval_json_structure(response: str) -> tuple[str, str]:
    """JSON 구조 유효성 체크."""
    # 응답에서 JSON 블록 추출 (markdown fence 포함)
    json_match = re.search(r'```(?:json)?\s*([\s\S]*?)```', response)
    text = json_match.group(1).strip() if json_match else response.strip()

    try:
        parsed = json.loads(text)
        return "pass", f"Valid JSON: {type(parsed).__name__}"
    except json.JSONDecodeError as e:
        return "fail", f"Invalid JSON: {e}"


def eval_python_syntax(response: str) -> tuple[str, str]:
    """Python 코드 구문 유효성 체크."""
    # 응답에서 Python 코드 블록 추출
    code_match = re.search(r'```(?:python)?\s*([\s\S]*?)```', response)
    text = code_match.group(1).strip() if code_match else response.strip()

    try:
        ast.parse(text)
        return "pass", f"Valid Python syntax ({len(text.splitlines())} lines)"
    except SyntaxError as e:
        return "fail", f"Python syntax error: {e}"
```

### 3.6 Quantization Diff

동일 프롬프트 + 동일 모델 베이스 + 다른 양자화(backend/quantization) 간 응답 비교.

```python
def compute_quant_diff(responses: list[dict]) -> dict:
    """동일 프롬프트에 대한 양자화 간 응답 비교.

    Args:
        responses: [{"model_name": "gemma3-1b-INT4", "response": "..."}, ...]

    Returns:
        {"pair": "INT4 vs INT8", "match_ratio": 0.85, "diff_summary": "..."}
    """
    if len(responses) < 2:
        return {}

    # 간단한 토큰 레벨 일치율
    from difflib import SequenceMatcher
    a = responses[0]["response"]
    b = responses[1]["response"]
    ratio = SequenceMatcher(None, a.split(), b.split()).ratio()

    return {
        "pair": f"{responses[0]['model_name']} vs {responses[1]['model_name']}",
        "match_ratio": round(ratio, 3),
        "a_length": len(a),
        "b_length": len(b),
    }
```

**사용 시나리오**: 같은 Gemma 1B 모델의 INT4 vs INT8 버전을 test_config.json에 등록하고 동일 프롬프트로 테스트. 양자화로 인한 품질 열화를 수치화.

### 7.2 핵심 로직 흐름

```python
def main():
    args = parse_args()
    db = get_connection()
    config = load_config()  # test_config.json에서 max_tokens 조회

    # 1. 미검증 results 조회
    rows = fetch_pending(db, args)
    # SELECT r.id, r.response, r.status, r.output_token_count,
    #        p.prompt_text, p.category, p.ground_truth, p.eval_strategy,
    #        m.model_name
    # FROM results r
    # JOIN prompts p ON r.prompt_id = p.id
    # JOIN models m ON r.model_id = m.id
    # WHERE r.validation_status IS NULL  (--force 시 제거)

    # 2. 검증
    results = []
    for row in rows:
        result = validate(row, config)
        results.append(result)

    # 3. DB 업데이트
    if not args.dry_run:
        update_validation(db, results)

    # 4. 양자화 diff (옵션)

        print_quant_report(diffs)

    # 5. 리포트

def validate(row, config) -> ValidationResult:
    # error status → skip
    if row.status == "error":
        return ValidationResult(row.id, "skip", {"checks": {}, "eval": None})

    # sanity checks
    checks = {
        "empty_response": len(row.response.strip()) == 0,
        "truncated": is_truncated(row, config),
        "gibberish": check_gibberish(row.response),
    }

    # empty → fail 즉시
    if checks["empty_response"]:
        return ValidationResult(row.id, "fail", {"checks": checks, "eval": None})

    # deterministic eval
    eval_result = None
    if row.eval_strategy == "deterministic":
        status, detail = eval_math(row.response, row.ground_truth)
        eval_result = {"strategy": "deterministic", "result": status, "detail": detail}
    elif row.eval_strategy == "deterministic_with_fallback":
        status, detail = eval_containment(row.response, row.ground_truth)
        eval_result = {"strategy": "deterministic_with_fallback", "result": status, "detail": detail}
    elif row.eval_strategy == "structural":
        if row.category == "code":
            status, detail = eval_python_syntax(row.response)
        else:
            status, detail = eval_json_structure(row.response)
        eval_result = {"strategy": "structural", "result": status, "detail": detail}

    # 최종 status 결정
    final_status = determine_status(checks, eval_result)
    return ValidationResult(row.id, final_status, {
        "checks": checks,
        "eval": eval_result,
        "quant_diff": None,
    })


def determine_status(checks: dict, eval_result: dict | None) -> str:
    """sanity check + eval 결과를 종합하여 최종 status 결정."""
    # eval이 fail이면 fail
    if eval_result and eval_result["result"] == "fail":
        return "fail"
    # eval이 uncertain이면 uncertain
    if eval_result and eval_result["result"] == "uncertain":
        return "uncertain"
    # sanity check에 경고가 있으면 warn
    if checks.get("truncated") or checks.get("gibberish"):
        return "warn"
    # eval이 pass이면 pass
    if eval_result and eval_result["result"] == "pass":
        return "pass"
    # eval 없으면 (none 카테고리) sanity만 통과 → pass
    return "pass"
```
